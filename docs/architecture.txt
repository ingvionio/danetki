================================================================
  ГЕНЕРАТОР ДАНЕТОК — АРХИТЕКТУРА ПРОЕКТА
================================================================

Данетка — это загадка, где есть открытая часть (то что даётся
игроку) и закрытая часть (то до чего он должен догадаться,
задавая вопросы с ответами "да" или "нет").

Система автоматически парсит реальные истории из интернета,
с помощью ИИ разбивает их на открытую и закрытую часть,
и выдаёт пользователю готовые данетки через REST API.

----------------------------------------------------------------
  КОМАНДА — 5 СЕРВИСОВ, 5 ЧЕЛОВЕК
----------------------------------------------------------------

  #1  API Gateway      Go
  #2  Auth Service     C#
  #3  Puzzle Service   C#
  #4  AI Worker        C#
  #5  Parser Service   Python

----------------------------------------------------------------
  ПОЛЬЗОВАТЕЛЬСКИЙ ПУТЬ
----------------------------------------------------------------

1. Пользователь отправляет HTTP-запрос на API Gateway (:8000)
2. Gateway проверяет JWT-токен (кроме /auth/*)
3. Gateway узнаёт адрес нужного сервиса через Consul
4. Gateway проксирует запрос по gRPC нужному сервису
5. Сервис отвечает, Gateway возвращает ответ клиенту

Основные сценарии:

  POST /auth/register   — регистрация
  POST /auth/login      — логин, получить JWT токен
  GET  /puzzle/random   — получить случайную данетку
  GET  /puzzle/:id      — получить конкретную данетку
  POST /parser/start    — запустить парсинг (фоновая задача)
  GET  /parser/status   — узнать статус парсинга

----------------------------------------------------------------
  КАК СЕРВИСЫ ОБЩАЮТСЯ
----------------------------------------------------------------

Два вида взаимодействия:

  gRPC (синхрон) — когда нужен ответ прямо сейчас.
  Используется везде где Gateway обращается к сервисам,
  и когда AI Worker отправляет готовую данетку в Puzzle Service.

  Kafka (асинхрон) — когда задача долгая и клиент не ждёт.
  Используется только между Parser и AI Worker.
  Parser кладёт историю в топик stories.raw,
  AI Worker забирает, обрабатывает через LLM (10-15 сек),
  и отправляет результат в Puzzle Service по gRPC.

Схема потока данных:

  Клиент
    ↓ HTTP REST
  API Gateway (Go)
    ↓ gRPC            ↓ gRPC              ↓ gRPC
  Auth Service     Puzzle Service      Parser Service
    ↓ SQL              ↓ SQL               ↓ NoSQL
  PostgreSQL        PostgreSQL           MongoDB
  (auth_db)         (puzzle_db)       (сырые истории)
                        ↑ gRPC
                    AI Worker (C#)
                        ↑ Kafka (stories.raw)
                    Parser Service

----------------------------------------------------------------
  ИНФРАСТРУКТУРА
----------------------------------------------------------------

  Docker Compose    — все сервисы запускаются одной командой
  Consul            — реестр сервисов, Gateway узнаёт у него
                      адреса сервисов вместо хардкода
  Kafka             — брокер сообщений между Parser и AI Worker
  PostgreSQL        — реляционная БД для Auth и Puzzle
  MongoDB           — документная БД для сырых историй Parser

----------------------------------------------------------------
  ОПИСАНИЕ КАЖДОГО СЕРВИСА
----------------------------------------------------------------

  #1 — API GATEWAY (Go)
  ---------------------
  Единственная точка входа в систему снаружи.
  Клиент знает только один адрес — Gateway.
  Все остальные сервисы закрыты внутри Docker-сети.

  Что делает:
  - Принимает HTTP-запросы от клиента
  - Проверяет JWT-токен на всех маршрутах кроме /auth/*
  - Узнаёт адрес нужного сервиса через Consul
  - Проксирует запрос по gRPC нужному сервису
  - Возвращает ответ клиенту

  Своей базы данных нет.

  Технологии: Go, gRPC-клиент, Consul HTTP API


  #2 — AUTH SERVICE (C#)
  ----------------------
  Отвечает за регистрацию, логин и проверку токенов.

  Что делает:
  - Регистрирует пользователя (хэширует пароль, сохраняет в БД)
  - Выдаёт JWT-токен при успешном логине
  - Проверяет валидность токена по запросу от Gateway

  gRPC методы:
    Register(email, password, username) → token, user_id
    Login(email, password)              → token, user_id
    Validate(token)                     → valid, user_id

  База данных: PostgreSQL (auth_db)
  Таблица users: id, email, password_hash, username, created_at

  Технологии: C#, ASP.NET, Entity Framework, BCrypt, JWT


  #3 — PUZZLE SERVICE (C#)
  ------------------------
  Хранит готовые данетки и выдаёт их по запросу.
  Получает данетки от AI Worker по gRPC.

  Что делает:
  - Выдаёт случайную данетку (open_part, без hidden_part)
  - Выдаёт конкретную данетку по ID
  - Принимает и сохраняет готовую данетку от AI Worker

  Важно: клиент никогда не получает hidden_part.
  Открытая часть (open_part) — то что видит игрок.
  Закрытая часть (hidden_part) — хранится в БД, не отдаётся.

  gRPC методы:
    GetRandomPuzzle()                         → puzzle_id, open_part, source_url
    GetPuzzleById(puzzle_id)                  → puzzle_id, open_part, source_url
    SavePuzzle(open_part, hidden_part, url)   → puzzle_id, success

  База данных: PostgreSQL (puzzle_db)
  Таблица puzzles: id, open_part, hidden_part, source_url, created_at

  Технологии: C#, ASP.NET, Entity Framework


  #4 — AI WORKER (C#)
  -------------------
  Фоновый воркер без HTTP-интерфейса.
  Забирает сырые истории из Kafka, отправляет в LLM,
  сохраняет готовую данетку через Puzzle Service.

  Что делает:
  - Подписывается на Kafka топик stories.raw
  - Получает историю, формирует промпт для LLM
  - Отправляет запрос в OpenAI API (GPT-4o-mini)
  - LLM возвращает open_part и hidden_part
  - Отправляет результат в Puzzle Service по gRPC

  Kafka: консьюмер топика stories.raw
  gRPC: клиент Puzzle Service (метод SavePuzzle)
  Своей базы данных нет — stateless воркер.

  Это место где реализована Saga:
  если AI Worker упал во время обработки — история
  остаётся в Kafka и будет обработана при перезапуске.

  Примерный промпт для LLM:
    "Раздели эту историю на две части:
     открытую (интригующее начало без развязки)
     и закрытую (объяснение что произошло на самом деле).
     Верни JSON: { open_part: '...', hidden_part: '...' }"

  Технологии: C#, Confluent.Kafka, OpenAI SDK, gRPC-клиент


  #5 — PARSER SERVICE (Python)
  ----------------------------
  Парсит сайты с необычными историями,
  сохраняет сырой текст и публикует в Kafka.

  Что делает:
  - Принимает команду запуска от Gateway по gRPC
  - Ходит на сайты с историями (httpx + BeautifulSoup)
  - Фильтрует подходящие истории (достаточно длинные,
    не дубликаты, содержат неожиданную развязку)
  - Сохраняет сырой текст в MongoDB
  - Публикует сообщение в Kafka топик stories.raw

  gRPC методы:
    StartParsing(limit) → message
    GetStatus()         → queued, processed

  Kafka: продюсер топика stories.raw
  Формат сообщения:
    {
      "story_id":   "uuid",
      "text":       "Полный текст истории...",
      "source_url": "https://...",
      "parsed_at":  "2025-01-15T10:30:00Z"
    }

  База данных: MongoDB (коллекция stories)
  Документ: story_id, text, source_url, parsed_at, status

  Технологии: Python, FastAPI, httpx, BeautifulSoup, Kafka, MongoDB

----------------------------------------------------------------
  ОБЩИЕ КОНТРАКТЫ (contracts/)
----------------------------------------------------------------

В корне репозитория лежит папка contracts/ с .proto файлами
и JSON-схемами. Все читают оттуда, никто не придумывает
форматы самостоятельно.

  contracts/
  ├── auth.proto          ← #1 и #2
  ├── puzzle.proto        ← #1, #3, #4
  ├── parser.proto        ← #1 и #5
  └── kafka/
      └── story.raw.json  ← #4 и #5

Генерация кода из .proto:
  Go:     protoc --go_out=.
  C#:     dotnet-grpc или protoc --csharp_out=.
  Python: python -m grpc_tools.protoc
